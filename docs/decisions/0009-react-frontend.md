# ADR 0009: React Frontend Migration

## Status

Accepted 2026-07-06.

## Context

ADR 0004 originally defaulted to modular vanilla ES modules unless a specific
framework need appeared. The app has now reached the point where that need is
known rather than speculative:

- the next shell work replaces a settings-window/control-sidebar app with a
  routed sidebar workspace model
- Preset Modes, Autopilot, Pattern Library, model management, voice controls,
  localization, and migration/setup will add more shared interactive UI
- Phase 14's pattern authoring surface is expected to need complex local editing
  state, reusable controls, and testable component boundaries

Deferring React until the authoring surface would make the project rewrite a
larger UI twice: once in vanilla modules now, then again in React later. That is
the maintainability cost this rewrite is meant to avoid.

The existing release constraint still matters: MagicHandy should run as a Go
binary with embedded web assets. A user must not need Node, npm, Vite, or a dev
server to run the app.

## Decision

Migrate the browser UI to React now, before implementing the sidebar-navigation
shell from `docs/ui-navigation-redesign.md` and `docs/ui-design-guidelines.md`.

Use React as a build-time frontend implementation, not a runtime service:

- Recommended stack: Vite + React + TypeScript.
- The React build emits static files that Go embeds and serves.
- Node/npm are development and CI dependencies only.
- Runtime releases remain a Go binary with embedded static assets.
- Keep CSS token driven and offline. Do not use CSS-in-JS, external fonts,
  external icon CDNs, or network-fetched assets.

## State Model Rules

React does not change the product state model.

- Backend snapshots/events remain authoritative for motion, transport,
  controller ownership, settings, prompt sets, memory, diagnostics, and model
  provider status.
- React state may hold local UI-only state: focused route, pending command IDs,
  input drafts, open popovers, optimistic pending labels, and editor dirty state.
- React state must not recreate a second motion engine, controller lease model,
  transport status model, or settings store.
- Optimistic UI is allowed only when visibly pending and reconciled with the next
  backend state update.
- Emergency Stop must remain a direct, always-mounted control path. It must not
  depend on the active route component being mounted.

## Build And Embedding Shape

The intended implementation shape is:

```text
web/
  package.json
  package-lock.json
  index.html              # Vite entry, not directly embedded as the app shell
  src/
    main.tsx
    App.tsx
    api/
    state/
    shell/
    routes/
    components/
    styles/
  dist/                   # generated by npm run build; embedded by Go
```

`web/assets.go` should embed the built output, for example `web/dist`, and return
an `fs.FS` rooted at that directory. Existing Go asset tests should be rewritten
to assert the generated app shell and critical hooks rather than old hand-written
files.

The exact output path can differ if implementation needs it, but the invariant is
fixed: `go run ./cmd/magichandy` serves static built assets without requiring
Node.

## Testing Requirements

The React migration PR should add frontend checks in CI:

- `npm ci`
- `npm run build`
- a fast component/unit test command, preferably Vitest + Testing Library
- existing Go checks still pass, including `go test ./...` and
  `CGO_ENABLED=0 go build ./cmd/magichandy`

At minimum, React tests should cover:

- Stop is rendered outside routed workspaces and remains present on Chat,
  Preset Modes, Pattern Library, and Settings.
- Status bar is status-only and does not contain Stop or live quick controls.
- Backend-loss state shows the persistent banner and locks backend-required
  controls.
- Read-only controller state disables command controls but leaves Stop available.
- Settings routes are real routes, not dialog overlays.
- Reduced-motion and accessible labels are preserved for the visualizer/status
  surfaces.

## Consequences

Positive:

- Avoids implementing the sidebar shell and later pattern editor twice.
- Gives the UI clear component ownership before workspaces multiply.
- Makes the upcoming Pattern Library and authoring UI more maintainable.
- Enables focused component tests around safety-critical UI invariants.

Negative:

- Adds Node/npm to development and CI.
- Adds a build step before Go embedding.
- Requires care to avoid React state becoming a duplicate backend state model.
- Existing `web/assets_test.go` source-fragment tests must be replaced with
  build-output and component-level assertions.
